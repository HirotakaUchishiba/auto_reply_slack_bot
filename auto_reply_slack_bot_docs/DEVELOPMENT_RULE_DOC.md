# Slack AIアシスタントBot 開発ルールブック

  
  

## 序文：本ルールブックの目的と基本原則

  
  

### 本ルールブックの目的

  

本ドキュメントは、Slack AIアシスタントBotアプリケーションの開発に関わるすべての技術者にとって、唯一の権威ある「信頼できる情報源（Single Source of Truth）」として機能します。本書に記載された規約、プロセス、およびベストプラクティスへの準拠は、プロジェクトに参加するすべてのメンバーに義務付けられます。

本ルールブックの主目的は、プロジェクトの技術設計仕様書 1 およびディレクトリ構成提案書 1 に詳述されている、高度なアーキテクチャ設計思想と構造的決定を、開発チームが日々の業務で実践可能な、明確かつ一貫性のある、強制力のある一連の規則へと翻訳することにあります。これにより、開発プロセス全体の品質、セキュリティ、保守性、および一貫性を保証します。

  

### 憲法としての中核設計原則

  

本プロジェクトにおけるすべての技術的判断は、場当たり的なものであってはなりません。それらは、技術設計仕様書で定義された4つの中核設計原則という、揺るぎない思想的基盤に基づいている必要があります 1。これらの原則は、本ルールブック全体を貫く「憲法」として機能し、個々の規則の背後にある「なぜ」を理解するための根拠となります。開発者は、単に規則に従うだけでなく、その背後にある設計思想を理解し、体現することが求められます。

1. セキュリティ・バイ・デザイン (Security-by-Design)  
    セキュリティは、開発プロセスの最終段階で追加される機能ではなく、設計の初期段階からすべてのコンポーネントに組み込まれるべき必須要件です。データプライバシーとセキュリティへの配慮を最優先事項と位置づけ、PIIデータの厳格な保護、脅威モデリング、そしてすべてのAWSリソースにおける最小権限の原則の徹底を義務付けます 1。
    
2. オブザーバビリティによる運用エクセレンス (Operational Excellence through Observability)  
    システムはデプロイ後に「ブラックボックス」であってはなりません。問題発生時に迅速な原因特定と解決を可能にするため、システムの内部状態を外部から観測できる「オブザーバビリティ（可観測性）」を確保します。構造化ロギング、分散トレーシング、そして技術的・ビジネス的メトリクスの監視とアラートをシステムのコア設計に不可分に組み込みます 1。
    
3. 回復力とスケーラビリティ (Resilience and Scalability)  
    システムは、構成要素の一部で障害が発生した場合でも、可能な限り正常に機能し続ける必要があります。これには、体系的なエラーハンドリング、指数バックオフを用いたリトライ戦略、そしてデータ損失を防ぐためのデッドレターキュー (DLQ)の実装が含まれます 1。
    
4. API・アズ・コントラクト (API-as-Contract)  
    システム内外のすべてのインターフェースは、厳格な「契約（コントラクト）」として扱われます。UIコンポーネントの完全なJSON構造の定義、外部API連携部分のカプセル化、そして明確なデータモデルの定義により、コンポーネント間の統合不全のリスクを根本的に排除します 1。

## 第1章：バージョン管理とブランチ戦略

  
  

### 1.1 リポジトリ構成：関心の分離の物理的具現化

  

本プロジェクトのリポジトリ構成は、単なるファイルの整理整頓ではなく、「関心の分離」という重要な設計原則を物理的に強制するためのフレームワークです 1。アプリケーションロジックとインフラストラクチャ定義が混在するような、密結合で脆弱なシステムにつながる一般的なアンチパターンを構造的に防止します。

以下に定義されるディレクトリ構成は、すべての開発者によって厳格に遵守されなければなりません。

  
  
  

auto_reply_slack-bot/  
├──.github/         # CI/CDと自動化ロジック  
├──.vscode/         # 共有エディタ設定  
├── docs/            # プロジェクトドキュメント  
├── scripts/         # 開発・運用補助スクリプト  
├── src/             # Pythonアプリケーションソースコード  
├── terraform/       # Infrastructure as Code (Terraform)  
├── tests/           # テストコード  
├──.gitignore  
├── Makefile         # 開発タスク自動化  
├── README.md  
└── requirements.txt  
  

各ディレクトリの責務は以下の通りです。

- .github/: GitHub Actionsを用いたCI/CDパイプライン定義など、すべての自動化ロジックを格納します。アプリケーションコードやインフラストラクチャコードから自動化の関心事を完全に分離します 1。
    
- docs/: アーキテクチャ決定記録（ADR）や運用ランブックなど、プロジェクトの補足ドキュメントを配置する専用の場所です 1。
    
- scripts/: ローカル開発やCI/CDパイプラインで使用されるヘルパースクリプト（例：Lambda関数のパッケージング）を格納します 1。
    
- src/: Pythonアプリケーションのビジネスロジックとインターフェースに関するコードのみを格納します。「何を」（アプリケーションロジック）と「どのようにどこで」（インフラストラクチャ）を厳密に分離するための核心的な領域です 1。
    
- terraform/: すべてのAWSリソースを管理するInfrastructure as Code（IaC）を格納します。アプリケーションロジックからの完全な分離が必須です 1。
    
- tests/: ユニットテスト、結合テスト、E2Eテストを含む、すべてのテストコードを格納します。srcディレクトリの構造をミラーリングすることで、コードとテスト間の対応を明確にします 1。
    

  

### 1.2 ブランチモデル：Git Flowに基づく規律ある開発

  

本プロジェクトでは、安定したリリースと効率的な並行開発を実現するため、Git Flowに基づいた厳格なブランチモデルを採用します。このモデルは単なる慣習ではなく、本番環境への意図しない変更を防ぎ、品質を保証するための重要なリスク管理戦略です。

  

#### デフォルトブランチ

  

リポジトリのデフォルトブランチは、必ずdevelopに設定されなければなりません。developブランチは、常に最新の開発状況を反映する統合ブランチとして機能し、すべてのフィーチャーブランチやバグフィックスブランチのマージ先となります。

  

#### 本番ブランチ

  

mainブランチは、本番環境にデプロイ済み、またはデプロイ準備が完了した、完全に安定したコードのみを保持する神聖なブランチです。mainブランチへの直接のコミットは固く禁じられています。すべての変更は、developブランチからmainブランチへのプルリクエストを通じてのみ反映されなければなりません。

このdevelopとmainの分離は、開発と本番の間に明確な「エアロック」を設けることを意図しています。CI/CDパイプラインは、developブランチへのマージをトリガーにstaging環境へデプロイし、mainブランチへのマージをトリガーにprod環境へデプロイするように構成されています。これにより、すべての変更が本番環境に到達する前に、stagingという本番に近い環境で統合・検証されるプロセスが強制され、mainブランチが常に安定した状態であることが保証されます。

  

### 1.3 ブランチ命名規則

  

すべてのブランチは、その目的とトレーサビリティを明確にするため、以下の命名規則に厳格に従わなければなりません。<issue-id>は、GitHub IssuesやJiraなどの課題管理システムにおけるチケット番号に対応している必要があります。

- フィーチャーブランチ: 新機能の開発に使用します。
    

- feature/<issue-id>-short-description
    
- 例: feature/TICKET-123-add-user-profile-modal
    

- バグフィックスブランチ: developブランチで発生したバグの修正に使用します。
    

- bugfix/<issue-id>-short-description
    
- 例: bugfix/TICKET-124-fix-modal-submission-error
    

- ホットフィックスブランチ: 本番環境（mainブランチ）で発生した緊急のバグ修正に使用します。
    

- hotfix/<issue-id>-short-description
    
- 例: hotfix/TICKET-125-critical-pii-leak-patch
    

  

### 1.4 コミットメッセージ規約

  

すべてのコミットメッセージは、Conventional Commits specificationに準拠しなければなりません。これにより、Gitの履歴が人間と機械の両方にとって読みやすくなり、変更内容の理解、CHANGELOGの自動生成、セマンティックバージョニングの自動化が容易になります。

コミットメッセージのフォーマットは以下の通りです。

  
  
  

<type>(<scope>): <subject>  
  
[optional body]  
  
[optional footer]  
  

- type: feat（新機能）、fix（バグ修正）、docs（ドキュメント）、style（コードスタイル）、refactor（リファクタリング）、test（テスト）、chore（ビルドプロセスなど）など。
    
- scope: 変更の範囲を示すコンポーネント名（例: api, pii, dynamodb）。
    
- subject: 変更内容を簡潔に説明する、50文字以内の短い要約。
    

例: feat(api): add endpoint for user profile retrieval

  

### 1.5 プルリクエスト・プロセス

  

すべてのコード変更は、プルリクエスト（PR）を通じてdevelopブランチ（およびmainブランチ）にマージされなければなりません。直接のプッシュは禁止です。

- PRの必須要件:
    

1. PRのタイトルと説明は、変更の目的と内容が明確に理解できるように記述し、関連する課題管理システムのチケットへのリンクを必ず含めます。
    
2. CI/CDパイプラインによって実行されるすべてのチェック（静的解析、ユニットテスト、Terraformプランなど）が成功していること。CIチェックが失敗したPRはマージできません。
    
3. 少なくとも1名の他のチームメンバーによるコードレビューと承認を受けていること。セルフマージは原則として禁止です。

## 第2章：コーディング規約とベストプラクティス

  

本章では、srcディレクトリ内のPythonアプリケーションコードと、terraformディレクトリ内のInfrastructure as Codeに適用される、具体的かつ強制力のあるコーディング規約を定義します。これらの規約は、コードの品質、可読性、保守性、そしてセキュリティを保証するための基盤です。

  

### 2.1 Pythonアプリケーションコード規約 (src/ ディレクトリ)

  

srcディレクトリ内のすべてのコードは、以下の規約に従う必要があります。これらの規約は、前述の4つの中核設計原則をコードレベルで具現化するものです。

  

#### 静的解析

  

すべてのPythonコードは、CI/CDパイプラインによって強制される静的解析チェックをパスしなければなりません。開発者は、コードをコミットする前にローカルでこれらのチェックを実行することが強く推奨されます。

- スタイルチェック: flake8を用いて、PEP 8に準拠した一貫性のあるコードスタイルを維持します 1。
    
- 型チェック: mypyを用いて、型ヒントに基づく静的な型安全性を保証します。これにより、実行時エラーを未然に防ぎ、コードの自己文書化を促進します 1。
    

  

#### 構造化ロギング

  

アプリケーション内のすべてのログ出力は、src/utils/logger.pyで定義された中央集権的なロガーファクトリを通じて行われなければなりません 1。

- 必須事項: このロガーはAWS Lambda Powertools for Pythonライブラリを利用し、すべてのログエントリが構造化されたJSON形式で出力され、リクエストごとに一意な「相関ID（Correlation ID）」を含むことを保証します 1。
    
- 禁止事項: 標準のprint()関数や、構造化されていないloggingモジュールの直接使用は固く禁じられています。これは、「オブザーバビリティによる運用エクセレンス」原則を遵守するための譲歩不可能な要件です。
    

  

#### 例外処理

  

- カスタム例外: アプリケーション固有の例外（例：OpenAIClientError, PIIRedactionError）は、src/utils/exceptions.pyに集約して定義しなければなりません 1。これにより、エラーハンドリングの意図が明確になります。
    
- 回復力: 特に非同期処理を担うLambda関数では、一時的な障害からの回復力を高めるため、指数バックオフを用いたリトライ戦略を実装し、最終的に失敗したイベントはデータ損失を防ぐためにSQSデッドレターキュー（DLQ）に送信されるように設計されなければなりません 1。
    

  

#### PIIリダクション

  

個人識別情報（PII）の保護は、本アプリケーションにおける最重要のセキュリティ要件です。「セキュリティ・バイ・デザイン」原則の核心的な実践として、以下のルールが適用されます。

- 必須プロセス: ユーザーから提供されたテキストを外部サービス（特にOpenAI API）に送信するすべての関数は、送信前に必ずsrc/core/pii_redaction.py内のprocess_text_for_openai関数でテキストを匿名化しなければなりません 1。
    
- 必須プロセス: 外部サービスからの応答に含まれるプレースホルダーは、ユーザーに表示する前に必ずrehydrate_text関数を用いて元のPII情報に復元しなければなりません 1。
    
- 禁止事項: このPIIリダクション・サブシステムをバイパスするいかなる試みも、重大なセキュリティ規約違反と見なされます。
    

  

#### 内部コントラクト

  

「API・アズ・コントラクト」原則は、外部インターフェースだけでなく、アプリケーション内部のデータフローにも適用されます。

- データモデル定義: APIリクエスト/レスポンスのペイロードやDynamoDBに保存されるアイテムなど、アプリケーション内で扱われる主要なデータ構造は、すべてsrc/utils/models.py内でPydanticモデルを用いてスキーマとして定義されなければなりません 1。
    
- 利点: これにより、実行時におけるデータのバリデーション、型強制、そしてIDEによる強力な補完サポートが提供され、データ関連のバグを早期に発見し、コードの堅牢性と可読性を大幅に向上させます。
    

このsrcディレクトリの細分化された構造は、単なる整理整頓以上の意味を持ちます。それは、開発者を「成功の穴（pit of success）」に導くための認知的な足場（cognitive scaffolding）として機能します。例えば、データベースへの新しいクエリを追加したい開発者は、自然とsrc/data_accessディレクトリを探すことになります。UIのJSON定義を変更する必要がある場合は、src/slack_ui/viewsがその場所であることが自明です。このように、適切に設計されたディレクトリ構造は、開発者が正しい場所に正しいコードを書くことを容易にし、アンチパターン（例：Lambdaハンドラ内にビジネスロジックやデータアクセスロジックを混在させる）の発生を構造的に抑制します。これは、認知的な負荷を軽減し、設計原則をファイルレベルで暗黙的に強制する、極めて効果的なアプローチです。

  

### 2.2 Infrastructure as Code (Terraform) 規約 (terraform/ ディレクトリ)

  

terraformディレクトリ内のすべてのコードは、再現可能で、安全かつ保守性の高いインフラストラクチャ管理を実現するために、以下の規約に従う必要があります。

  

#### モジュール性

  

- 必須事項: すべてのAWSリソース（Lambda関数、IAMロール、DynamoDBテーブルなど）は、terraform/modules/ディレクトリ内で再利用可能な独立したモジュールとして定義されなければなりません 1。
    
- 責務: 各環境のルートモジュール（例: terraform/environments/dev/main.tf）の役割は、これらの汎用モジュールを組み合わせて環境全体を構成することに限定されます。ルートモジュール内に直接リソースを定義することは、原則として禁止です。これは、コードの重複を避けるDRY（Don't Repeat Yourself）原則を徹底するためです。
    

  

#### 環境分離

  

- 必須事項: Lambdaのメモリサイズ、DynamoDBのキャパシティユニット、ログレベルといった環境固有の設定値は、各環境の.tfvarsファイル（例: staging.tfvars, prod.tfvars）で変数として定義されなければなりません。
    
- 禁止事項: これらの値を.tfファイル内にハードコーディングすることは固く禁じられています。これにより、単一のコードベースから複数の環境を安全かつスケーラブルに管理することが可能になります。
    

  

#### バリデーション

  

- 必須事項: 開発者は、コードをコミットしてプルリクエストを作成する前に、ローカル環境でterraform fmtコマンドによるコードフォーマットの整形と、terraform validateコマンドによる構文検証を必ず実行しなければなりません。これらのチェックはCI/CDパイプラインでも強制されます 1。
    

  

#### 機密情報管理

  

- 禁止事項: OpenAI APIキーやデータベースのパスワードといったいかなる機密情報も、Gitリポジトリ内に（たとえ.tfvarsファイル内であっても）保存してはなりません。
    
- 必須事項: すべての機密情報は、AWS Secrets ManagerまたはAWS Systems Manager Parameter Storeで安全に管理し、Terraformコードからはデータソースとして参照する形式を取らなければなりません 1。これは、「セキュリティ・バイ・デザイン」原則の基本的な要件です。

## 第3章：品質保証とテスト戦略

  

アプリケーションの品質と回復力は、偶然の産物ではなく、意図的な設計と厳格なテスト戦略によってのみ達成されます。本プロジェクトでは、設計仕様書で定義された多層的なテスト戦略を採用し、コードの品質をあらゆるレベルで保証します 1。

  

### 3.1 テストの責務

  

コードは、それを検証するための包括的なテストが伴って初めて「完了」と見なされます。新機能の実装やコードの修正を行った開発者は、その変更に対応するテストコードを記述する責任を負います。テストのないコードは、マージを許可されません。

  

### 3.2 ユニットテスト (tests/unit/)

  

ユニットテストは、個々の関数やモジュールを分離された環境で検証し、ビジネスロジックの正確性を保証するための第一の防御線です。

- 対象: src/core（コアビジネスロジック）、src/data_access（データアクセスロジック）、src/utils（共通ユーティリティ）内のすべてのロジックは、網羅的なユニットテストによってカバーされなければなりません。
    
- 構造: tests/unit/ディレクトリの内部構造は、src/ディレクトリの構造を正確にミラーリングしなければなりません 1。例えば、  
    src/core/pii_redaction.pyに対するテストは、tests/unit/core/test_pii_redaction.pyに配置します。これにより、コードとテストの対応関係が即座に理解でき、テストされていないコードの発見が容易になります。
    
- モッキング: ユニットテストは、外部依存（特にAWSサービス）から完全に独立している必要があります。AWSサービスとのインタラクションは、motoライブラリを用いてモック化しなければなりません 1。ユニットテストが実際のAWSリソースに対してネットワークコールを行うことは固く禁じられています。
    

  

### 3.3 結合テスト (tests/integration/)

  

結合テストは、個々のユニットが正しく連携して動作することを検証する、より高いレベルのテストです。

- 目的: API Gatewayのエンドポイントへのリクエストから、Lambda関数の実行、DynamoDBへのデータ書き込み・読み込みに至るまで、デプロイされた複数のAWSサービス間の相互作用が期待通りに機能することを確認します 1。
    
- 実行環境: 結合テストは、CI/CDパイプラインの一部として、staging環境へのデプロイが成功した後に自動的に実行されます。本番環境（prod）に対して破壊的な可能性のあるテストを実行することは禁止です。
    
- 配置場所: これらのテストコードはtests/integration/ディレクトリに配置します 1。
    

  

### 3.4 エンドツーエンド (E2E) テスト (tests/e2e/)

  

エンドツーエンドテストは、ユーザーの視点からシステム全体のワークフローをシミュレートし、ビジネス要件が満たされていることを最終的に確認するテストです。

- シナリオ: 専用のテスト用Slackワークスペースを使用し、実際のユーザー操作（例：メッセージショートカットのクリック、モーダルへの入力と送信、ボットからの最終的な応答の検証）を自動化します 1。
    
- 目的: このテストは、フロントエンド（Slackクライアント）からバックエンドのすべてのコンポーネント、そして外部API（OpenAI）連携までを含む、完全なユーザージャーニーを検証します。
    
- 配置場所: これらのテストコードはtests/e2e/ディレクトリに配置します 1。

## 第4章：CI/CDパイプラインとデプロイメント規約

  

本プロジェクトでは、迅速かつ安全なデリバリーを実現するため、すべてのコードとインフラストラクチャの変更を自動化されたCI/CDパイプラインを通じて管理します。

  

### 4.1 パイプライン概要：唯一のデプロイ経路

  

すべてのコードおよびインフラストラクチャの変更は、.github/workflows/deploy.ymlで定義されたGitHub Actions CI/CDパイプラインを通じてのみ、各環境にデプロイされなければなりません 1。開発者のローカルマシンからの手動デプロイや、AWSコンソールを介した直接の変更は、緊急時を除き固く禁じられています。

このdeploy.ymlファイルは、単なる自動化スクリプト以上の存在です。それは、本プロジェクトの開発プロセス、ガバナンスモデル、そして品質保証戦略そのものをコードとして表現したものです。パイプラインの各ステージは、本ルールブックで定義された規約を機械的に強制する役割を果たします。

- Lint & Static Analysisジョブは、第2章のコーディング規約を強制します。
    
- Unit Testsジョブは、第3章の品質保証戦略を強制します。
    
- Terraform Planジョブは、インフラ変更に対する「適用前の計画」という安全メカニズムを強制します。
    
- Deploy to Productionジョブのブランチ条件（if: github.ref == 'refs/heads/main'）は、第1章で定義されたGit Flowブランチモデルを強制します。
    

このように、CI/CDパイプラインは、本ルールブックで定められたすべてのルールを生命あるものにするエンジンであり、開発ライフサイクル全体を駆動する中心的なメカニズムです。

  

### 4.2 環境戦略とTerraform Workspaces

  

本プロジェクトでは、安全な開発とテストを保証するため、以下の3つの独立した環境を維持します。これらの環境は、AWSアカウントレベルで分離することが強く推奨されます 1。

- dev (開発環境): 個々の開発者がフィーチャーブランチからデプロイし、分離された環境で機能開発とテストを行うための環境。
    
- staging (検証環境): developブランチの最新の状態を常に反映する、本番環境とほぼ同等の環境。結合テストと最終的な受け入れテストが実施されます。
    
- prod (本番環境): 実際にエンドユーザーが利用するライブ環境。mainブランチの安定したコードのみがデプロイされます。
    

これらの複数環境の状態管理は、単一のTerraformコードベースから、Terraform Workspacesを用いて行われなければなりません。これにより、コードの重複を排除し、各環境の状態（.tfstateファイル）を安全かつ独立して管理します。

  

### 4.3 Staging環境へのデプロイ

  

フィーチャーブランチやバグフィックスブランチからのプルリクエストがレビューされ、developブランチにマージされると、CI/CDパイプラインが自動的にトリガーされ、変更がstaging環境にデプロイされます。これにより、staging環境は常に最新の開発状況を反映し、継続的な統合とテストが可能になります。

  

### 4.4 本番環境へのデプロイ

  

staging環境での検証が完了し、リリース準備が整ったコードは、developブランチからmainブランチへのプルリクエストを作成することで、リリース候補となります。

このプルリクエストが承認され、mainブランチにマージされると、CI/CDパイプラインが再びトリガーされ、変更がprod環境に自動的にデプロイされます。設計仕様書に示されている通り、この本番デプロイステージには、意図しないリリースを防ぐため、GitHub Actionsのenvironment機能を用いた手動承認ステップを設けることが可能です。

## 第5章：セキュリティ開発ライフサイクル

  

セキュリティは、本プロジェクトのあらゆる側面に組み込まれるべき最優先事項です。「セキュリティ・バイ・デザイン」の原則に基づき、すべての開発者は以下のセキュリティ規約を遵守する責任を負います。

  

### 5.1 セキュアコーディング・チェックリスト

  

本番環境へのリリース前、およびすべてのプルリクエストにおいて、以下の項目が満たされていることを確認しなければなりません。これは、技術設計仕様書（第6.2章）で定義されたチェックリストを、開発者の日常業務における必須の規約として具体化したものです 1。

- [ ] PIIリダクションの徹底: OpenAI APIなどの外部サービスに送信されるすべてのユーザー入力に対し、PIIリダクション・サブシステムが有効であり、機能していることを確認する。
    
- [ ] 最小権限のIAM: すべてのIAMロールが、その機能に必要な最小限の権限のみを持つように構成されていることを確認する。ワイルドカード（*）の使用は厳しく制限する。
    
- [ ] DLQの設定: 非同期処理を行うすべてのLambda関数に、エラーハンドリング戦略の一環としてデッドレターキュー（DLQ）が正しく設定されていることを確認する。
    
- [ ] 保存データの暗号化: DynamoDBテーブルにおいて、保存時の暗号化（Encryption at Rest）が有効になっていることを確認する。
    
- [ ] 機密情報の安全な管理: OpenAI APIキーなどのシークレット情報が、コードや環境変数に直接記述されず、AWS Secrets Managerで管理されていることを確認する。
    
- [ ] 本番環境のロギングレベル: 本番環境（prod）にデプロイされるLambda関数で、デバッグレベル（DEBUG）の詳細なイベントログが無効化されていることを確認する。
    
- [ ] ドメイン認証（SES使用時）: オプション機能であるメール送信を使用する場合、送信元ドメインにSPF、DKIM、DMARCレコードが正しく設定・検証されていることを確認する。
    

  

### 5.2 依存関係管理

  

サードパーティライブラリに存在する脆弱性は、アプリケーション全体のセキュリティに対する重大な脅威となり得ます。

- 必須事項: プロジェクトの依存関係（requirements.txtに記載）は、GitHubのDependabotやSnykのような自動化ツールを用いて、既知の脆弱性がないか定期的にスキャンされなければなりません 1。
    
- 対応義務: 重大（Critical）または高（High）レベルの脆弱性が検出された場合、開発チームは本番環境へのデプロイ前に、その脆弱性を修正（ライブラリのアップデートまたは置換）する義務を負います。
    

  

### 5.3 IAMポリシーの規約

  

インフラストラクチャのセキュリティは、IAMポリシーの適切な管理にかかっています。

- 必須事項: terraform/modules/iam/ディレクトリで新しいIAMロールやポリシーを作成・変更する際は、常に最小権限の原則に従わなければなりません 1。
    
- 禁止事項: ActionやResourceにおける安易なワイルドカード（例: s3:*, Resource: "*"）の使用は、明確な技術的根拠があり、コードレビューで承認された場合を除き、固く禁じられています。権限は、必要なアクションとリソースに対してのみ、明示的に付与する必要があります。

## 付録：主要コマンドリファレンス

  

開発者体験の一貫性を保ち、一般的なタスクを簡素化するため、プロジェクトのルートディレクトリにMakefileが用意されています 1。開発者は、日常的な操作にこれらの標準化されたコマンドを使用することが推奨されます。

- 静的解析の実行:  
    コードのスタイルと型安全性をローカルで検証します。  
    Bash  
    make lint  
      
    
- ユニットテストの実行:  
    すべてのユニットテストをローカルで実行します。  
    Bash  
    make test-unit  
      
    
- Terraformプランの生成:  
    staging環境に対するインフラストラクチャの変更計画をプレビューします。  
    Bash  
    make tf-plan-staging  
      
    
- Lambdaパッケージの作成:  
    srcディレクトリのPythonコードを、AWS Lambdaにデプロイするためのzipパッケージにまとめます。  
    Bash  
    make package-lambda
